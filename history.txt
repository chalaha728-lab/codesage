1. INITIAL BUG REPORT
--------------------------------------------------------------------
User hit a repeating Gemini free-tier quota error:

    You exceeded your current quota... 429 Too Many Requests
    generativelanguage.googleapis.com/generate_content_free_tier_requests
    limit: 20, model: gemini-3.6-flash

Expected behavior: on quota error, rotate to the next configured API key.
Actual behavior: rotation was "supposed to" happen but didn't.

--------------------------------------------------------------------
2. BUG #1 — Telemetry gate silently blocked error-triggered rotation
--------------------------------------------------------------------
Found in extension/dist/extension.original.js, inside captureProviderFailure().

recordKeyError() (the function that actually rotates keys on error) was only
ever called from inside a block gated by:

    providerFailureTelemetryTurnGate.shouldCaptureStreamingFailure()

That gate allows at most ONE streaming-failure capture per "turn" — it was
designed to throttle telemetry, but recordKeyError() was nested inside it, so
after the first quota error in a turn, every subsequent 429 in the same turn
skipped rotation entirely.

Fix: moved the recordKeyError() call out from behind the telemetry gate so it
fires on every rotate-worthy failure; kept the gate around the telemetry
(captureProviderApiError) call only.

--------------------------------------------------------------------
3. BUG #2 — "Requests per key" threshold setting didn't persist
--------------------------------------------------------------------
User reported: set threshold to 3, save, navigate away, come back — reset to 2.

Root causes (three separate holes):
  - codesage_setRotation message handler never actually stored the
    codesage_threshold value sent from the webview.
  - getRotationState() (VQp) hardcoded threshold:2 in its return value,
    ignoring anything saved.
  - The SessionFactory call site that builds the key-rotation resolver
    hardcoded the literal 2 as the threshold argument.

Fix: added persisted secret storage key `codesage:threshold:{provider}`
(mirrors existing key-storage pattern), wired setRotation to save it,
getKeys to load/report it, and the resolver-creation call site to read the
stored value instead of the hardcoded 2.

--------------------------------------------------------------------
4. BUG #3 — RetryError swallowed the real HTTP status
--------------------------------------------------------------------
Even after fix #1, auto-rotation still silently no-op'd. Root cause: the
Vercel AI SDK's own internal retry wrapper (pya/fya) retries transient errors
a couple of times, then on final failure throws a new AI_RetryError that
wraps the real error in `.errors[]` / `.lastError`, with no top-level
`.status`/`.statusCode` of its own. The error-normalizer (qH) that feeds
recordKeyError only checked `o.status || o.statusCode || o.response?.status`
— none of which exist on the wrapper — so status came back undefined and
csIsRotateWorthyStatus(undefined) was always false.

Fix: qH's constructor now also unwraps `o.lastError` (and falls back to the
last entry of `o.errors[]`), checking `.status` / `.statusCode` /
`.response?.status` / `.cause?.status` on the inner error too.

Also added a message-text fallback (csIsRotateWorthyMessage) that matches
patterns like "quota", "rate limit", "too many requests", "resource_exhausted",
"429", "401", "403", "unauthorized", "invalid api key", "permission denied" —
so rotation still triggers even if a numeric status never survives whatever
wrapping layer produced the error.

--------------------------------------------------------------------
5. BUG #4 — Rotating the key didn't affect the already-running session
--------------------------------------------------------------------
Even with correct status detection, the user reported it "still stops" —
same key kept getting hit until they manually picked a different one in the
UI. Root cause: the resolver/API key is only read once, at session/model
construction time (inside buildSessionConfig / jfu). The only code path that
rebuilds an active session's model client is handleApiConfigurationChanged(),
which only fires when the PROVIDER ID itself changes (e.g. gemini→openai) —
rotating between two Gemini keys never changes the provider id, so that path
always bailed out and the live session kept using its originally-baked key.

Fix (session-level): captureProviderFailure() now checks whether
recordKeyError() actually rotated (it returns true/false), and if so, and the
affected provider is the currently active one, schedules a rebuild through
the same idle-aware queue the provider-switch path uses:

    this.sessionRebuilds.request("keyRotation", () =>
      this.providerChanges.restartActiveSessionForProviderChange());

This only applies once the session goes idle (so it won't interrupt an
in-flight generation) — which is exactly the moment right after a failed
turn.

--------------------------------------------------------------------
6. BUG #5 — Real per-request rotation (Portkey-style architecture)
--------------------------------------------------------------------
Looked at how Portkey's open-source AI Gateway (github.com/Portkey-AI/gateway)
implements reliability: fallback/rotation happens at the raw HTTP fetch
layer, inline, per request — on a 429/401/403/5xx it immediately retries the
SAME request against the next key/target before ever returning control to the
caller. It never depends on rebuilding a client or waiting for a new
turn.

Implemented the same pattern: added csMakeRotatingFetch(providerId,
originalKey, baseFetch), which wraps the fetch function passed into the SDK
client. On every actual HTTP call it:
  - substitutes the currently active rotated key into the request (generic,
    provider-agnostic: finds the originally-baked key string wherever it
    appears in headers/url and swaps in the new one — works regardless of
    auth scheme: Bearer header, x-goog-api-key, x-api-key, ?key= query
    param, etc.)
  - if the response is 401/403/429/5xx, rotates to the next key and retries
    the SAME request immediately, up to the number of configured keys
  - only returns to the SDK once it gets a success or exhausts all keys

Wired into buildSessionConfig() in place of the previously-static `fetch:Jc`.
This is now genuinely live, per-request, mid-turn rotation — matching
Portkey's own gateway architecture — independent of session-rebuild timing.

7. FEATURE — Manual key selector
--------------------------------------------------------------------
Added ability to manually pick which key is "active" from the API config
key list:
  - Backend: csSelectKey(providerId, keyId) — sets currentKeyIndex directly,
    resets requestCount, mirrors the key into the legacy secret slot so it's
    live immediately even with rotation off.
  - New codesage_selectKey message handler.
  - codesage_getKeys now returns codesage_activeKeyId so the webview knows
    which key is current.
  - Webview: each key row shows a highlighted border + green "● Active"
    badge for the current key; inactive keys get a "Use" button to switch
    immediately.

--------------------------------------------------------------------
8. FEATURE — Live key-rotation status widget (webview)
--------------------------------------------------------------------
Added an event-driven live status bar above Key Management:
  - Backend: csEmit()/csEventLog/csSetEventSink/csGetRotationLog — every
    rotation (count-based, error-triggered/live-fetch, or manual) is logged
    (last 25 events per provider, in-memory) and pushed to the webview in
    real time via a `codesage_keyRotationEvent` postMessage, in addition to
    being retrievable on demand via codesage_getKeys.
  - Webview: compact always-visible bar showing:
      - pulsing dot (green = rotation enabled) that flashes on live events
      - "Live key: <label>" — current active key, updates instantly
      - thin progress bar for requestCount / threshold
      - badge with total key count
    Expandable (chevron) event log below it, newest first, color-coded by
    reason:
      ⚡ HTTP <status> on "<from>" → "<to>" (attempt n/total)   — live/error rotation
      ↻  Rotated "<from>" → "<to>" after <threshold> requests  — count-based
      👆 Manually switched to "<to>"                            — manual selection
    No polling required — updates live off the pushed event.
